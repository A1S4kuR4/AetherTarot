# ADR 0001：采用单仓库（Monorepo）与文档先行的包边界

- 状态：Accepted
- 日期：2026-04-06
- 决策人：待补充

---

## 背景

AetherTarot Agent 不是一个单纯的聊天界面项目，而是一个包含以下部分的长期系统：

- 产品应用层
- Agent 编排层
- 领域规则层
- 知识编译层
- 安全与评测层
- 面向 Codex 的协作层

如果所有逻辑混在单一应用目录里，后续会很快出现以下问题：

- 提示词、规则、知识、评测和代码耦合严重
- 代码 Agent 难以判断修改边界
- 文档很难跟上实现变化
- 输出协议与安全规则容易漂移

---

## 决策

项目首版采用：

1. **单仓库（Monorepo）** 组织方式
2. **文档先行** 的目录结构
3. 将 `docs/` 作为项目真相层之一
4. 将 `knowledge/` 作为独立知识编译层
5. 将 `evals/` 作为独立质量治理层
6. 将复用逻辑逐步沉淀到 `packages/`
7. 首轮迁移只落地 `shared-types`、`domain-tarot`、`prompting` 与 `data/`
8. `apps/web` 作为当前唯一活跃应用，`/api/reading` 暂留在 `web` 内作为 BFF 路由
9. `prototype/` 冻结保留，仅作为历史参考，不进入 workspace

---

## 原因

### 正面原因

- 更适合长周期迭代
- 更适合多模块同时演化
- 更适合 Codex 这类代码 Agent 理解项目边界
- 更容易把知识、规则与评测外部化
- 更容易做 ADR 和文档治理

### 我们有意接受的代价

- 初期目录看起来更重
- 需要更严格的文档纪律
- 需要维护更清晰的模块职责边界

---

## 备选方案

### 方案 A：单应用仓库，文档最小化

优点：

- 快速启动
- 目录简单

缺点：

- 后期混乱风险高
- 不利于长期知识沉淀
- 不利于 Agent 协作

### 方案 B：一开始就多仓库拆分

优点：

- 物理隔离更清晰

缺点：

- 初期成本过高
- 跨仓文档和评测协作复杂
- 对当前阶段过度设计

---

## 后果

采用本决策后：

- 所有关键模块都需要先定义文档职责
- 后续新增模块应优先判断属于 `apps/`、`packages/`、`knowledge/`、`evals/` 中哪一层
- 对输出协议、安全边界、解释框架的改动应同步更新文档
