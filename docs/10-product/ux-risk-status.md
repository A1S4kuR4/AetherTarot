# UX 风险状态（UX Risk Status）

- `last_updated`: `2026-04-09`
- `owner`: `Codex`
- `scope`: `rolling UX risk status`

## 1. 文档目的

本文件用于持续回答一个更具体的问题：

> 当前版本的 Web 主流程还剩下哪些尚未关闭的 UX / 产品风险？

当前最新评审结论（`2026-04-09`）是：**没有完全解决。**

本文件作为持续更新的状态文档使用。当前记录显示：这轮改动显著改善了仪式感、页面节奏和结果页质感，但多数风险仍停留在“被缓解”而不是“被机制化关闭”的状态。

## 3. 总体判断

### 已明显缓解，但未闭环

- `R2` 抽卡像纯随机程序噪声
- `R4` 赛博塔罗感过强
- `R5` 留存不足
- `R9` 页面流程完整但节奏不成立
- `R11` 边界声明只在文档里强，产品内机制弱

### 仍然明显存在

- `R1` 核心价值感不清
- `R3` 迎合错觉
- `R6` 决策外包 / 情绪依赖入口
- `R7` 人格过于安全、温和、可预期
- `R8` 解释体系过强，用户价值感没有同步增强
- `R10` 结构化输出显模板化
- `R12` 本地持久化限制长期产品形态

---

## 4. P0 剩余风险

### R1. 核心价值感不清

**当前判断：仍然存在**

当前首页已经更像一个仪式入口，但用户第一眼仍然主要看到“提问 + 选牌阵 + 开始仪式”。

这意味着用户仍可能把产品理解为：

- 有塔罗皮肤的 AI 回答器
- 更讲氛围的生成器
- 用牌阵包装输出的问答工具

**为什么未被解决**

- 首页没有明确把“问题-牌阵-张力-历史回看”的独特链路讲成核心机制
- 牌阵选择仍更像参数选择，而不是阅读方式选择
- history 的长期价值没有前置为产品主卖点

**建议优先动作**

- 在首页前置说明“牌阵如何组织问题，而不是直接替你回答”
- 把“回看主题变化”和“阶段性自我记录”提升为主价值，而不是附属页面
- 给不同牌阵增加更明确的使用场景说明

### R2. 抽卡被感知为纯随机程序噪声

**当前判断：部分缓解，仍然存在**

这轮 UX 改动确实让抽牌不再像一次普通按钮点击。仪式页、洗牌动作、逐张抽取、揭示页都增强了体验层的成立感。

但从用户理解层看，风险没有被真正关闭。

**为什么未被解决**

- 抽牌逻辑仍然是直接随机洗牌、随机抽取、随机逆位
- 没有机制说明“为什么这不是瞎随机”
- 没有焦点校准或问题确认步骤来建立“被组织的随机”

**建议优先动作**

- 在抽牌前加入问题焦点校准
- 明确告诉用户：你选择的是阅读容器，不是操纵结果
- 在揭示页解释牌阵位置如何组织本次随机

### R3. 迎合错觉

**当前判断：仍然存在**

当前版本已经比“深背景输入 + 大段个性化输出”的形态更克制，因为前台只收一个问题输入。

但这并不等于风险已经消失。

**为什么未被解决**

- 结果页没有区分“牌面线索”和“系统推断”
- 综合段仍会直接围绕用户问题进行组织
- 当前前台没有任何地方帮助用户理解：哪些内容更接近牌阵支持，哪些内容属于解释扩展

**建议优先动作**

- 在结果页显式拆开“牌面线索 / 位置语义 / 综合推断”
- 控制对用户原话的复述密度
- 保留少量不完全顺着用户期待的张力点

### R4. 赛博塔罗感过强

**当前判断：部分缓解，仍然存在**

仪式页与揭示页已经明显降低了“直接出报告”的感觉，这是当前 UX 改动中最成功的部分之一。

但当用户进入最终解读页后，体验仍然很容易回到“结构化系统输出”的认知。

**为什么未被解决**

- reading 页依然是稳定字段分块
- 从 reveal 到 reading 的转折更多是视觉跳转，不是理解上的转折
- 不同牌阵在前台展开方式上的差异还不够大

**建议优先动作**

- 让 reveal 先给出整组牌的整体氛围和主张力
- 降低 reading 页的模块对称感
- 为不同牌阵设计不同的前台展开重心

## 5. P1 剩余风险

### R6. 用户把产品当作决策外包或情绪依赖入口

**当前判断：仍然明显存在**

当前已有免责声明和 `safety_note`，但它们主要仍是文案型、后置型边界，而不是行为约束。

**为什么未被解决**

- 没有针对重复同类追问的提醒
- 没有针对高频使用的节制机制
- 关系确认、重大决策类问题仍可直接进入正常流程

**建议优先动作**

- 对短时间重复同类问题加入节制提醒
- 对重大决策类问题加入“先补现实信息”的前置摩擦
- 把“这不是替你决定”从声明做成交互机制

### R5. 留存不足，history 还不是复访理由

**当前判断：部分缓解，仍然存在**

历史页已经比纯列表更完整，阅读页也新增了“反思手记”。

这说明产品已经开始从“结果存档”向“个人记录”迈了一步。

但这还不足以形成复访理由。

**为什么未被解决**

- history 仍主要是单次结果列表
- 手记仍挂靠在单次 reading 上，而不是阶段性主题上
- 没有形成“这段时间你一直在面对什么”的持续关系感

**建议优先动作**

- 支持同一问题的后续补记和变化回看
- 聚合同类主题，而不只是并列历史条目
- 提供“从上次到现在发生了什么变化”的复访入口

### R10. 结构化输出过稳，显模板化

**当前判断：仍然明显存在**

当前 reading 页几乎是 output schema 的直接前台映射。

这保证了工程稳定，但也让用户更容易觉得：每次只是在固定模块里替换内容。

**为什么未被解决**

- 主题、逐牌、综合、指引、延伸的顺序稳定且同构
- 不同牌阵与问题类型在前台几乎共享同一阅读骨架
- guidance / follow-up 的呈现差异不够大

**建议优先动作**

- 保留 schema，但允许不同牌阵采用不同前台编排
- 某些问题类型优先呈现整组气候，再进逐牌
- 避免所有 reading 都以同样节奏展开

### R11. 边界声明只在文档里强，产品内机制弱

**当前判断：部分缓解，仍然存在**

首页、侧边栏、结果页与 safety override 已经把边界搬进了产品。

这是正确方向，但还没有形成真正的用户行为引导。

**为什么未被解决**

- 边界大多停留在说明文案
- 高风险问题仍以结果页补充声明为主
- 产品前半程缺少“可问什么 / 不适合问什么”的前置引导

**建议优先动作**

- 在提问前给出可问 / 不适合问的提示
- 对高风险问题在提交前做一次边界确认
- 更早地提供现实支持与专业帮助导向

### R7. 产品人格过于安全、温和、可预期

**当前判断：仍然明显存在**

UX 风格已经更有气氛，但生成逻辑层仍高度依赖固定问题类型镜头、固定 guidance 与固定追问模板。

**为什么未被解决**

- 文本展开方式的变化幅度有限
- 长期使用时容易形成熟悉模式
- 安全与克制正在吞掉人格锐度

**建议优先动作**

- 让不同问题类型有不同语气张力
- guidance 不要每次都落回类似的安抚句式
- 在安全前提下保留少量“阻力感”

### R8. 解释体系过强，用户价值感没有同步增强

**当前判断：仍然明显存在**

这轮 UX 主要改善的是界面与仪式感，并没有直接解决“用户为什么会持续回来”的核心价值问题。

**为什么未被解决**

- 当前体验仍主要围绕解释结构展开
- 用户长期价值尚未转化成产品机制
- 团队仍容易继续优化内部解释质量，而不是外部价值显性

**建议优先动作**

- 每次新增能力先问：它是否提升复访价值
- 优先做长期关系与阶段连续性，而不是继续堆内部完整度
- 减少只对内部结构有益、对用户体感无增益的展示

### R9. 页面流程完整，但节奏仍未完全成立

**当前判断：明显改善，尚未闭环**

当前主流程已经形成“提问 -> 仪式 -> 揭示 -> 解读”的链路，这比之前更接近真正的阅读体验。

但 `reveal` 还没有真正承担“理解转折点”的角色。

**为什么未被解决**

- reveal 更像展示卡面，而不是帮助用户建立整组牌的理解框架
- reading 第一屏仍然以结构化内容为主
- history 页仍偏归档而非复盘

**建议优先动作**

- 让 reveal 输出“整组牌先给你的整体感受”
- 把 reading 第一屏从“结构”转成“主张力”
- 在 history 里聚合阶段主题，而不是只展示单次条目

### R12. 本地持久化策略限制长期产品形态

**当前判断：未受本轮 UX 改动影响**

这轮 UX 改动没有改变持久化边界。当前历史与手记仍然依赖本地存储。

这意味着它仍然会在中期限制：

- 跨设备连续性
- 长期陪伴感
- 用户关系资产积累
- 自然演化为长期反思工具的能力

**建议优先动作**

- 在架构层尽早定义未来的长期连续性模型
- 即使短期不做服务端持久化，也要先定义演进边界

---

## 6. 当前建议

如果只按近期收益排序，下一轮 UX / 产品工作建议优先聚焦三件事：

1. 把首页与抽牌流程从“好看”推进到“可理解”  
   重点解决 `R1` 与 `R2`

2. 把 reading 页从“稳定报告”推进到“有张力的阅读体验”  
   重点解决 `R3`、`R4`、`R10`

3. 把 history 从“回放列表”推进到“阶段性自我记录”  
   重点解决 `R5` 与 `R12`
