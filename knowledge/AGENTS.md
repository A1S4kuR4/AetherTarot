# 知识层工作协议 (Knowledge Layer Schema)

## 1. 角色与使命

本文件是 `knowledge/` 目录的操作规范。当你（Agent）在此目录下进行操作时，你扮演的是**“塔罗知识编辑”**的角色。
本目录（知识库）的最终使命是：**为 AetherTarot 解读引擎提供可靠、结构化、可长期维护的塔罗领域知识。**
你需要遵循 Karpathy 的 LLM-Wiki 架构思想，逐步将松散的原始素材编译成体系化的知识网络。

## 2. 三层架构说明

本项目知识管理遵循严格的三层架构职责划分：

| 目录/文件 | 职责定位 | 可变性约束 |
| :--- | :--- | :--- |
| **`raw/`** | 原始素材层。包含外部书籍、文献、记录的原始文档。 | **不可变 (Immutable)**：只读、只增、不改。 |
| **`wiki/`** | 编译知识层。LLM（你）将 `raw/` 提取综合后产出的结构化 markdown 页面集合。 | **活文档 (Living)**：Agent 主力编写，人类审阅共创，随时重构。 |
| **`AGENTS.md`** <br> **`index.md`** <br> **`log.md`** | Schema 与治理层。定义编译规则、页面覆盖度索引与变更日志。 | **结构化更新**：根据本文档（AGENTS.md）定义的规则执行并更新状态。 |

## 3. Wiki 本体论（页面分类）

`wiki/` 目录下的知识应被精确分类并存放在对应的子目录中：

1. **`wiki/major-arcana/`**：22 张大阿卡纳牌（大牌），每张牌一个单独的文件。
2. **`wiki/minor-arcana/`**：56 张小阿卡纳牌（小牌）。内部按花色划分为四个子目录：`wands/`、`cups/`、`swords/`、`pentacles/`。每张牌一个单独的文件。
3. **`wiki/concepts/`**：塔罗主题概念页。例如四元素理论、数字学基础、宫廷牌性格体系、卡巴拉生命之树对应等。
4. **`wiki/spreads/`**：牌阵知识页。侧重对应牌阵的解读策略、位置张力分析等领域知识（区别于 `data/spreads/` 中用于运行时的纯数据结构）。

## 4. 页面模板

在创建或重构 `wiki/` 页面时，必须包含以下结构和 Frontmatter：

### 4.1 牌义页模板（Major/Minor Arcana）
```md
---
title: "牌的中文名 (English Name)"
card_id: "全局唯一标识符，如 the-fool, ace-of-cups"
arcana: "major | minor"
suit: "wands | cups | swords | pentacles | null (大牌为 null)"
number: Int (如 0, 1, 14; 宫廷牌可为对应的数值或特定标识)
element: "fire | water | air | earth | null"
sources: ["78W", "YAT", "CTB"] # 来源简称数组
last_updated: YYYY-MM-DD
---

# [标题]

## 1. 核心象征与视觉意象
描述牌面主要视觉元素及其象征意义。

## 2. 核心牌义（正位）
总结正位时的主要解释方向。

## 3. 核心牌义（逆位）
总结逆位时的力量转变或流向。如果某些流派不采用逆位，请说明。

## 4. 常见问题类型倾向
- **关系/情感**：...
- **事业/发展**：...
- **自我反思**：...

## 5. 多源视角与争议
记录不同经典资料之间对此牌说明的差异或补充角度（请标注来源，如 `[CTB]` 认为...）。

## 6. 关联与交叉引用
展示与之有较强呼应或对称关系的牌页或概念页链接。
```

## 5. 命名与格式约定

- **文件名**：必须使用**英文小写、kebab-case**格式（例：`the-fool.md`, `two-of-wands.md`, `four-elements.md`）。
- **内容语言**：正文主体使用**中文**。牌名在文档标题或首次出现时，须标注 `中文名 (English Name)`。
- **交叉引用**：引用其他 wiki 页面时，必须使用**相对路径 Markdown Link**（例：`详见 [愚者](../major-arcana/the-fool.md)`）。不要使用孤立词汇，建立实质链接。
- **来源引用**：知识点引用须标注简写标识（例：`[来源: 78W]`、`[来源: CTB]`）。

## 7. Ingest（摄入）工作流

当指令要求你将新的 `raw` 资料摄入到 `wiki/` 时，执行以下步骤：
1. 若资料未注册，先更新本文件中的《来源注册表》。
2. 阅读/提取目标资料中的特定牌义、概念或牌阵内容。
3. 对应创建或更新 `wiki/` 下的相关页面，按照“页面模板”整合信息。如果现有页面已有其他来源内容，请保留并以“多源视角”整合，切勿全部覆盖抹杀历史信息。
4. 更新 `index.md` 中的覆盖度追踪矩阵。
5. 在 `log.md` 中增加一条 Ingest 变更记录。

## 8. Lint（质量检查）工作流

当被要求执行 Lint 检查时，验证以下内容：
1. **覆盖度**：`index.md` 矩阵是否准确反映了物理文件存在状态；78 张牌是否都有实体落地。
2. **完整性**：抽查 wiki 页面，检查 Frontmatter 和必填二级 Section 是否齐备缺失。
3. **一致性**：检查页面间的相对路径交叉引用链接是否破损（Dead links）；检查引用的 `Source ID` 是否在注册表内。
4. **陈旧度**：标记明显缺乏特定资料来源支撑的核心页面。

## 9. 冲突处理策略

不同流派及作者对牌义和概念会存在差异，你的处理原则是：
1. **拥抱多元**：不可强行统一差异，请在 `多源视角` 章节分列各方观点。
2. **框架为主**：核心牌义提取时，需参照 `docs/20-domain/interpretation-framework.md` 里的框架，抽取能够辅助反思的实质内容进行串联。
3. **标明来源**：存在明显分歧时，紧贴分歧点使用 `[来源: XXX]` 标注。
4. **搁置非业务决策**：碰到机制性矛盾（如某书坚决反对使用逆位），由 `docs/` 层做决策，此处只客观记录各书对逆位的解释逻辑。

## 10. 与 `docs/` 目录的边界墙

极度重要！请严格遵守以下知识隔离：
- **`knowledge/wiki/` 负责回答：“这牌/概念/牌阵究竟是什么含义？”**（纯客观领域知识）
- **`docs/20-domain/` 负责回答：“系统应该怎么调用这些知识来做解答？”**（解读框架、组合规则）
- **`docs/50-safety/` 负责回答：“系统绝对不能输出什么？”**（底线、危机响应）
**禁止**在 `wiki/` 页面里写入产品的行为限制或安全免责声明（如“系统不能做医疗建议”），这是越界行为。

## 11. `index.md` 维护规则

`knowledge/index.md` 包含了分类总目录和塔罗牌覆盖度矩阵（78-Card Checklist）。每次创建新的牌页面时，必须将矩阵中相应的 `[ ]` 更新为 `[x]` 并链接至该文件。

## 12. `log.md` 维护规则

任何对 `wiki/` 发起的结构性写入/变更，都必须在 `knowledge/log.md` 添加条目。格式请参考该文件头部的示例说明，包含日期、操作类型（ingest/lint/update）、影响范围及操作主体（Agent ID 或名称）。
