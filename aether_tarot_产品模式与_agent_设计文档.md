# AetherTarot 产品模式与 Agent 设计文档

## 1. 文档目的

本文档基于当前对 AetherTarot 的连续讨论，整理并统一以下内容：

1. AetherTarot 应采用的产品塔罗模式
2. 更接近真实塔罗咨询的流程设计
3. 不同塔罗师 Agent 的分层设计方案
4. LLM 的选择原则、分层调用思路与推荐组合
5. 产品风险与落地建议

本文档的目标不是把 AetherTarot 做成“会说塔罗话的聊天机器人”，而是将其设计为一个更接近真实咨询体验、同时具备产品可控性与工程可落地性的数字塔罗系统。

---

## 2. AetherTarot 的核心产品定位

### 2.1 不应做成什么

AetherTarot 不应被设计为以下任一形态：

- 纯随机抽卡后立即生成一段“像塔罗”的文案
- 依赖大量用户预填信息，再把这些信息包装成占卜话术
- 通过过强的情绪迎合与模糊表述来制造“被看见”的错觉
- 以“神秘感”为核心卖点，却缺乏结构化、可解释与现实边界

这些方向会导致以下问题：

- 明显的“赛博塔罗感”
- 用户产生“系统只是在顺着我说”的迎合错觉
- 抽牌沦为装饰，真正起作用的是问卷与话术重组
- 难以建立长期信任与产品辨识度

### 2.2 应做成什么

AetherTarot 更适合的定位是：

**一个以牌为起点、以对话为校正、以反思式整合为核心的数字塔罗咨询系统。**

它应当同时具备以下特征：

- 牌先说话，而不是用户先把所有信息说完
- 问题导向，而不是命盘导向
- 两阶段解读，而不是一次性直接宣判
- 结构化输出，而不是纯情绪化长文
- 保持非断言式表达与现实校验，而不是替用户做决定

换句话说，AetherTarot 的核心不是“预测”，而是：

**帮助用户围绕一个当下问题，看清当前局势、核心张力、阻碍、选择代价与行动建议。**

## 3. 更接近真实塔罗咨询的产品模式

### 3.1 不存在唯一的“真正塔罗模式”

现实中的塔罗大致可分为三类：

1. 娱乐式 / 单次解牌  
   用户提问，抽牌，塔罗师直接解释，流程短，体验快，但容易浅表化。

2. 咨询式 / 对话式解牌  
   先明确问题，再抽牌，再根据牌面追问，再综合解读。这种方式更接近真实咨询。

3. 反思式 / 陪伴式塔罗  
   重点不是给出唯一答案，而是帮助用户梳理情绪、关系、处境与选择。

AetherTarot 最适合选择第 2 类与第 3 类的结合：

**咨询式流程 + 反思式输出。**

### 3.2 最推荐的核心模式：两阶段解读

最接近真实塔罗、同时也最适合产品化的模式不是“一问一答式立即生成”，而是：

**两阶段解读模式**

#### 阶段一：抽牌后的独立初读

系统在用户给出问题范围后，先完成以下动作：

- 收束问题范围
- 完成抽牌
- 基于问题与牌面，输出独立于用户补充信息的初步判断
- 给出 1 到 3 个基于牌面矛盾的解释方向
- 提出 1 到 2 个定向追问

这一阶段的作用是：

- 让用户感到“牌本身先产生了内容”
- 避免系统变成纯问卷式套话
- 为第二阶段建立一个不可轻易漂移的牌面主轴

#### 阶段二：用户补充后的整合深读

用户回答追问后，系统再进行第二阶段生成：

- 收束解释方向
- 将牌面主题与现实语境对齐
- 输出综合判断、核心阻碍、盲点、行动建议
- 根据需要给出 sober check / 风险提醒 / 观察信号

这一阶段的目标不是“重写答案”，而是：

**在保留第一阶段主轴的前提下，用用户补充信息去校正解释空间。**

### 3.3 真实塔罗体验的关键原则

#### 原则一：牌先说 60%，用户补 40%

好的塔罗体验不是用户先说 80%，系统再包装成塔罗语言；而是牌先提出方向性判断，再由用户补充现实语境来做收束。

#### 原则二：追问必须锚定牌面矛盾

追问不是为了套资料，而是为了缩小解释空间。

正确的追问方式应围绕：

- 拉扯点
- 防御点
- 阻滞点
- 行动与承诺的失衡
- 情绪与现实的冲突

而不是泛泛地问：

- 你最近是不是很焦虑
- 你是不是遇到了某个人
- 你是不是工作不顺

#### 原则三：最终输出不是“宣判”，而是“分层解释”

更真实的塔罗咨询不应该只给结论。最终输出至少应包含：

1. 牌意层：这组牌在讲什么
2. 情境层：它在用户当前处境中对应什么
3. 行动层：用户接下来可以怎么面对
4. 边界层：哪些结论不能被过度神化，哪些决定需要现实校验

## 4. 抽卡前是否需要用户个人信息

### 4.1 塔罗与星座 / 八字的不同

塔罗本质上不是“出生图谱系统”。

与占星、星座、本命盘、八字不同，塔罗通常不依赖以下信息作为核心输入：

- 出生年月日
- 出生时刻
- 出生地点
- 八字命盘结构

星座与八字关注的是“先天结构”，而塔罗更关注：

- 当前问题
- 当前关系
- 当前处境
- 当前选择
- 当前阻碍与趋势

因此，AetherTarot 的主流程不应默认收集出生信息。

### 4.2 抽卡前真正需要的，是问题语境

在更接近真实塔罗的流程里，抽卡前所需的是最小必要的问题语境，而不是命理建档信息。

推荐收集：

- 问题领域：感情 / 工作 / 关系 / 决策 / 状态
- 具体想看什么：趋势 / 阻碍 / 建议 / 对方态度 / 行动方向
- 时间范围：近期 / 一个月 / 三个月等
- 相关对象：自己 / 两人关系 / 某个机会 / 某项选择
- 当前阶段：刚开始 / 持续中 / 已分开 / 正在犹豫

### 4.3 不建议默认收集的内容

为了降低迎合感与“信息回填”感，不建议在主流程中默认收集：

- 出生年月日
- 星座
- 八字
- MBTI
- 完整情感经历
- 详细家庭背景
- 大量自我叙述

### 4.4 可选的扩展模式

如果未来产品想做“混合灵感模式”，可以提供独立选项：

- 纯塔罗模式：只看问题语境与牌面
- 混合灵感模式：可选加入星座 / 生日 / 元素象征等，作为风格层补充

但应明确标注：

**这属于体验增强或混合占卜，而不是纯塔罗核心流程。**

## 5. AetherTarot 的标准咨询流设计

### 5.1 建议的完整流程

AetherTarot 的标准产品流程建议设计为：

1. 用户提出大致问题范围
2. 系统进行最小问题澄清
3. 用户确认本次问题边界
4. 系统抽牌
5. 系统输出第一阶段独立初读
6. 系统基于牌面提出少量定向追问
7. 用户回答追问
8. 系统输出第二阶段整合深读
9. 系统附加现实提醒与 sober check

### 5.2 各步骤说明

#### 步骤一：提出问题范围

用户通常不会一开始就给出特别清晰的问题。系统应允许模糊输入，如：

- 我想问感情
- 我最近工作很乱
- 我不知道该不该离开一段关系

#### 步骤二：最小问题澄清

系统应只做必要的收束，不应做成长问卷。

#### 步骤三：确认边界

系统需要明确本次 reading 回答的是哪一个问题，而不是把多个问题混在一起。

#### 步骤四：抽牌

抽牌是咨询成立的重要仪式节点。无论视觉呈现还是动画设计，都应让用户感受到“进入阅读”的时刻。

#### 步骤五：第一阶段独立初读

初读应包含：

- 核心张力
- 主要方向判断
- 可能的解释支路
- 当前最值得澄清的矛盾点

#### 步骤六：定向追问

追问数量应严格控制。默认建议：

- Lite：0 到 1 个
- Standard：1 到 2 个
- Deep：2 到 3 个

#### 步骤七：用户回答

用户回答后，系统不应推翻第一阶段，而应基于第一阶段收束解释。

#### 步骤八：第二阶段整合深读

整合深读应输出：

- 主题判断
- 当前处境解释
- 核心阻碍
- 用户盲点
- 行动建议
- 观察信号

#### 步骤九：现实提醒与 sober check

在涉及重大决策、关系冲突、精神压力、强依赖倾向时，需要强化：

- 非断言式表达
- 风险提醒
- 现实可验证性
- 不替代现实决策与专业帮助

## 6. 不同塔罗师 Agent 的设计原则

### 6.1 为什么适合做多个塔罗师 Agent

AetherTarot 非常适合做“不同咨询深度的塔罗师 Agent”。

原因在于：

- 项目本身采用两阶段结构，很适合做工作流深度差异
- 用户对塔罗产品的需求层次不同
- 不同成本模型可以映射到不同档次产品
- 多 Agent 可以增强产品丰富度与商业化空间

### 6.2 不该做成什么样的多 Agent

不建议做成大量仅靠文风区分的人设型塔罗师，例如：

- 高冷塔罗师
- 毒舌塔罗师
- 温柔姐姐塔罗师
- 灵性塔罗师

原因包括：

- 容易变成换皮产品
- 用户会察觉底层逻辑其实一样
- 评测困难，难以判断问题来自模型、流程还是人设
- 过强的戏剧化人设可能损害信任

### 6.3 正确方向：共享内核，不同 Profile

应采用：

**一个统一 Tarot Core + 多个 Agent Profile**

共享核心模块：

- 问题收束
- 抽牌逻辑
- 牌面初读
- 追问生成
- 综合解读
- 安全检查
- sober check
- 结构化输出协议

Agent 之间的差异应主要体现在：

- 追问轮数
- 初读深度
- 解读层次
- 行动建议强度
- 现实校验强度
- 输出篇幅
- 调用模型等级

## 7. 推荐的塔罗师 Agent 分层方案

### 7.1 快速塔罗师（Lite Reader）

#### 定位

轻量、快速、低门槛，适合首次体验与轻咨询。

#### 适合场景

- 用户只想快速看一个倾向
- 问题不复杂
- 用户不想回答太多问题

#### 流程特点

- 允许弱两阶段，甚至只做极简追问
- 追问数量少
- 输出较短
- 更偏启发而非深咨询

#### 输出重点

- 当前倾向
- 核心张力
- 简短建议

### 7.2 标准塔罗师（Standard Reader）

#### 定位

最核心的默认模式，也是最接近真实咨询的标准版本。

#### 适合场景

- 大多数普通咨询
- 用户愿意接受一次定向追问
- 需要完整但不过重的解读

#### 流程特点

- 标准两阶段
- 1 到 2 个基于牌面的定向追问
- 完整主题总结与建议

#### 输出重点

- 主题判断
- 情境映射
- 核心阻碍
- 行动建议

### 7.3 深度塔罗师（Deep Reader）

#### 定位

高投入、高沉浸、高客单价版本，更接近咨询型塔罗师。

#### 适合场景

- 复杂关系
- 职业转折
- 高情绪密度问题
- 用户希望获得更深入的结构化梳理

#### 流程特点

- 更细致的问题收束
- 多条假设支路
- 2 到 3 个高依据追问
- 更长的整合分析

#### 输出重点

- 当前局势
- 核心阻碍
- 用户盲点
- 行动路径
- 未来观察信号

### 7.4 清醒塔罗师（Sober Reader）

#### 定位

不是更神秘，而是更克制、更稳健，是 AetherTarot 最有辨识度的特色模式之一。

#### 适合场景

- 重大决策
- 强依赖倾向
- 容易情绪化解读的问题
- 高风险关系或现实压力情境

#### 流程特点

- 结构上接近 Standard 或 Deep
- 强化 sober check 与现实提醒
- 降低断言感
- 强化替代性解释与现实约束

#### 输出重点

- 当前处境分析
- 风险提醒
- 现实边界
- 不建议直接把 reading 当成决策依据的提醒

### 7.5 第一版上线建议

第一版不建议同时上线太多 Agent。

推荐优先上线：

1. 快速塔罗师
2. 标准塔罗师
3. 清醒塔罗师

理由：

- 已足够覆盖轻咨询、标准咨询、风险控制三类核心需求
- 更方便评测与调优
- 能先验证两阶段模式是否被用户接受
- 能验证清醒模式是否真的提升信任感

深度塔罗师可作为第二阶段新增模式。

## 8. LLM 选择原则

### 8.1 AetherTarot 对模型的真实要求

AetherTarot 不是普通聊天产品，因此模型评估维度应区别于通用聊天：

1. 中文表达是否细腻、自然、非模板化
2. 是否能稳定输出结构化 JSON
3. 是否能在两阶段流程中保持主轴一致，不轻易漂移
4. 是否能做到非断言式表达与边界控制
5. 是否能兼顾情绪理解与现实提醒
6. 成本是否适合不同档次产品分层

因此，模型选择不应只看“文采”或“是否会说玄学话术”，而应同时评估：

- 结构稳定性
- 语义一致性
- 中文咨询质感
- 安全与 sober 输出能力
- 成本

### 8.2 为什么不建议纯本地主模型作为主测试基线

在当前阶段，不建议将本地中小模型作为 AetherTarot 的主质量评测基线。

原因在于：

- 你当前要测的是咨询体验与结构质量，而不只是能否出字
- 本地中小模型可能能输出内容，但不一定能稳定完成结构、边界与细腻表达
- 它们更适合接口联调、前端冒烟、回归测试，而不是定义产品体验上限

因此更适合采用：

**付费 API 作为主评测基线，本地模型作为辅助开发与回归工具。**

## 9. 中国 LLM API 推荐路线

### 9.1 优先推荐的厂商方向

从 AetherTarot 当前需求看，中国 API 更推荐以下方向：

#### 第一梯队：Qwen、Kimi

适合原因：

- 中文表达能力较强
- 更适合做结构化输出与多阶段工作流
- 容易承接中高质量咨询体验
- 适合作为标准档与高级档核心模型

#### 第二梯队：DeepSeek

适合原因：

- 成本较低
- 适合批量测试、回归与轻量级结构任务
- 可承担低成本流程节点

#### 第三梯队：Doubao

适合原因：

- 可作为流量层、批量层与成本敏感场景的补充
- 更适合做入口层、路由层或大规模服务层

### 9.2 推荐的总体判断

如果希望优先追求工程稳定性与分层清晰度：

**首选 Qwen 体系单供应商分层。**

如果希望在质量与成本之间做更灵活平衡：

**Qwen + Kimi + DeepSeek 的组合更适合。**

## 10. LLM 分层调用方案

### 10.1 不同功能由不同模型承担

AetherTarot 非常适合“按功能分配模型”，而不是所有任务都交给同一个模型。

建议采用四层模型职责：

#### 第 1 层：入口判断层

负责：

- 问题分类
- 领域判断
- 时间范围归一化
- 风险预判
- 是否需要 sober check 预标记

要求：便宜、快、稳定。

适合模型：

- Qwen Flash
- Doubao Lite
- DeepSeek-chat

#### 第 2 层：主解读生成层

负责：

- 第一阶段初读
- 第二阶段整合深读
- 主要结构化输出

要求：

- 中文细腻
- 结构稳定
- 主轴一致
- 咨询感强

适合模型：

- Qwen Plus
- Qwen Max
- Kimi 高质量模型

#### 第 3 层：安全复核层

负责：

- 审核断言风险
- 校正高依赖倾向
- 检查是否需要 sober note
- 过滤高风险表达

要求：

- 成本较低
- 格式稳定
- 规则执行可靠

适合模型：

- DeepSeek-chat
- Qwen Flash
- Doubao Lite

#### 第 4 层：文风润色层（可选）

负责：

- 改善可读性
- 微调表达温度
- 优化 follow-up questions 的自然度

注意：

- 不应改写牌义主轴
- 不应推翻前面阶段的结论

### 10.2 面向用户的产品分层

比起简单的 basic / pro，更推荐把产品分层定义为：

- Lite Reading
- Standard Reading
- Deep Reading
- Sober Reading

每种档次的差异不只在模型，而在工作流深度。

## 11. 推荐的具体模型组合

### 11.1 方案 A：最稳单供应商方案

适合：

- 快速落地
- 统一接口
- 维护成本低

建议组合：

- Gate：Qwen Flash
- Standard 主解读：Qwen Plus
- Deep 主解读：Qwen Max
- Safety Review：Qwen Flash

优点：

- 供应商统一
- 分层清晰
- 接入与维护成本较低

### 11.2 方案 B：质量优先的多供应商方案

适合：

- 更关注中文质感上限
- 愿意维护多 Provider
- 追求成本 / 质量更细致平衡

建议组合：

- Gate：DeepSeek-chat
- Standard 主解读：Qwen Plus
- Premium / Deep 主解读：Kimi
- Safety Review：DeepSeek-chat 或 Qwen Flash

优点：

- 成本更灵活
- 高端档表达可拉高
- 便于做高低档明显区分

## 12. 架构与实现建议

### 12.1 架构原则

应采用：

**一个 Tarot Core + 多个 Agent Profile + 多 Provider 路由层**

这意味着：

- Tarot Core 定义统一流程
- Agent Profile 定义不同咨询深度与参数
- Provider Router 负责模型选择
- Safety Layer 独立执行边界控制

### 12.2 Agent Profile 可配置参数

建议每个 Agent 配置以下参数：

- analysisDepth
- maxFollowupQuestions
- initialReadGranularity
- actionAdviceLevel
- soberCheckStrength
- outputLength
- modelProfile
- toneProfile

### 12.3 输出结构建议

推荐统一结构化字段，包括：

- cards
- themes
- synthesis
- reflective_guidance
- follow_up_questions
- safety_note
- confidence_note
- sober_check
- presentation_mode

不同 Agent 共享字段定义，只调整字段深度与输出强度。

## 13. 产品风险与控制建议

### 13.1 迎合错觉风险

风险：

- 系统追问过多
- 用户提供信息过多
- 最终输出看起来像把用户原话换个说法

控制建议：

- 强制保留第一阶段独立初读
- 严格限制追问数量
- 追问必须锚定牌面矛盾

### 13.2 赛博塔罗风险

风险：

- 语言过于模板化
- 抽牌只是视觉装饰
- 输出缺少真实咨询结构

控制建议：

- 强化两阶段流程
- 把“牌先说话”变成核心规则
- 让初读具备真实独立判断

### 13.3 过度神化风险

风险：

- 用户把 reading 当成现实决策依据
- 系统输出过度断言

控制建议：

- 强化 sober 模式
- 对重大问题默认附加风险提醒
- 明确说明 reading 是辅助反思，不替代现实判断

### 13.4 多 Agent 换皮风险

风险：

- 只是 prompt 换口气
- 用户很快发现底层没有真正差异

控制建议：

- 差异落在流程与输出结构，而非纯文风
- 先做少量 Agent，验证后再扩充

## 14. 第一阶段实施建议

### 14.1 产品上先做什么

第一阶段建议优先实现：

1. 纯塔罗模式
2. 两阶段标准咨询流
3. 三个 Agent：快速 / 标准 / 清醒
4. 至少一套清晰的 sober check 规则
5. 一套统一的结构化输出协议

### 14.2 模型上先怎么接

建议优先采用以下策略：

- 日常开发与接口联调：本地模型或低成本模型
- 主体验评测：付费高质量 API
- 正式产品初版：先采用单供应商或双供应商方案，不要一开始过度复杂

### 14.3 评测上先看哪些指标

建议重点观察：

- 用户是否感到“牌先说话”
- 是否感到被理解，而非被迎合
- 不同 Agent 是否真的有可感知差异
- sober 模式是否提升信任感
- 输出是否稳定、结构是否一致

## 15. 最终结论

AetherTarot 最有潜力的发展方向，不是做成一个会说玄学话术的 AI，而是做成一个：

**以两阶段解读为核心、以反思式咨询为本质、以不同深度塔罗师 Agent 为产品分层、以多模型路由为底层能力的数字塔罗系统。**

其中最关键的原则有四个：

1. 牌先说话，而不是用户先把全部信息说完
2. 两阶段解读是产品真实性的核心机制
3. 多 Agent 要做咨询深度差异，而不是换皮人设差异
4. LLM 应按功能分层，不应由单一模型承担全部职责

如果按照这一方向推进，AetherTarot 将更有机会同时解决以下问题：

- 降低赛博塔罗感
- 避免迎合错觉
- 提升产品可信度
- 丰富用户层次体验
- 为后续商业化建立清晰分层
